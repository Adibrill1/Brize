# חוות דעת ארכיטקטונית — Brill.ai Nomad Dashboard

**תאריך:** 2026-07-04
**מבוסס על:** מסמכי ה-Handoff (Project Requirements + Handoff: Nomad Dashboard)

---

## תמצית

החזון נכון והיקף הפרויקט ריאלי לבנייה עצמאית: כלי Map-First לניהול מסע של משתמשת אחת, עם עדיפות לעריכה ידנית מלאה. עקרונות העיצוב (Manual Override על כל החלטה אוטומטית, מסך פיקוד אחד, Offline) הם בדיוק הנכונים לסוג כזה של מוצר.

עם זאת, יש **בעיה ארכיטקטונית קריטית אחת** ו**שני דגלים אדומים של היתכנות** שכדאי לפתור לפני שכותבים שורת קוד:

1. **שלושה מקורות אמת מתחרים** — JSON מקומי, Google Sheets ו-Google Calendar, כולם ברי-עריכה, עם דרישת Offline. זהו מתכון לאובדן נתונים וקונפליקטים.
2. **Smart Outreach אוטומטי** — לפלטפורמות הרלוונטיות (Airbnb, TrustedHousesitters) אין API ציבורי, ופנייה אוטומטית מפרה את תנאי השימוש שלהן.
3. **ייבוא Wishlist מ-Google Maps** — אין API רשמי לרשימות שמורות; אפשרי רק כייבוא ידני-למחצה.

כל השלושה פתירים — הפתרונות מפורטים בהמשך.

## 1. הבעיה המרכזית: מקור אמת יחיד

### המצב במסמך
הסטאק המוצע מגדיר בפועל שלושה מאגרי נתונים שכולם ניתנים לכתיבה:

- JSON/GeoJSON מקומי (בשביל Offline)
- Google Sheets ("כ-Database לעריכה ידנית")
- Google Calendar (סנכרון דו-כיווני)

בנוסף, Make.com כותב גם הוא — כלומר **ארבעה כותבים** על אותם נתונים.

### למה זה מסוכן
- סנכרון דו-כיווני בין שני מקורות הוא בעיה קשה בפני עצמה; בין שלושה — עם עריכות Offline שנצברות בתור — כמעט בלתי אפשרי לייצב. התסמין הצפוי: עריכה ידנית שנעלמת כי אוטומציה דרסה אותה. זה פוגע בדיוק בערך המרכזי של המוצר (Editability).
- ל-Google Sheets אין טרנזקציות ואין נעילות. שני כותבים במקביל (האפליקציה + Make.com) יגרמו ל-Race Conditions.
- מגבלות קצב של Sheets API (60 בקשות/דקה למשתמש) הופכות אותו לצוואר בקבוק כ-Backend חי של PWA.

### ההמלצה: Local-First
- **מקור האמת היחיד:** מסד נתונים מקומי ב-PWA — IndexedDB (למשל דרך Dexie.js). כל הישויות (תחנות, לינות, מקטעי נסיעה, הוצאות) חיות שם כרשומות מובנות. זה גם פותר את דרישת ה-Offline "בחינם".
- **Google Calendar כהשתקפות (Projection), לא כמסד נתונים:**
  - דחיפה חד-כיוונית של אירועים מהאפליקציה ליומן.
  - קריאה חזרה מצומצמת בעזרת Incremental Sync Tokens, רק לאירועים שהאפליקציה מזהה כשלה (תיוג עם `extendedProperties`).
  - כלל קונפליקט פשוט וקשיח: **עריכה אנושית אחרונה מנצחת; אוטומציה לעולם לא דורסת עריכה ידנית בלי אישור.**
  - מטא-דאטה (מחיר, חניה, קישורים) נשאר במסד המקומי — ביומן רק זמן, כותרת ומזהה.
- **Google Sheets יורד בדרגה לייצוא/דוחות** (חד-כיווני מהאפליקציה החוצה), או מוסר לגמרי. "עריכה ידנית" מקבלת מענה טוב יותר בממשק המפה עצמו — שזה ממילא עמוד התווך של המוצר.
- **גיבוי בין מכשירים:** אם מדובר במכשיר אחד — ייצוא JSON תקופתי ל-Google Drive מספיק. אם צריך כמה מכשירים — Backend דק (Supabase) עדיף בהרבה על Sheets.

## 2. דגלי היתכנות

### 2.1 Smart Outreach
- ל-Airbnb אין API ציבורי; ל-TrustedHousesitters ודומיו אין בכלל. פנייה אוטומטית דרך הפלטפורמות מפרה תנאי שימוש ועלולה לגרום לחסימת חשבון — סיכון שאסור לקחת באמצע מסע של שנתיים.
- **המלצה: חצי-אוטומציה.** המערכת מייצרת טיוטת פנייה מותאמת אישית (זה החלק שה-AI באמת טוב בו), והשליחה נעשית ידנית בלחיצה. פניות מבוססות אימייל (מארחים פרטיים, House Sitting ישיר) אפשר לאוטמט מלא דרך Gmail API. זה גם עקבי עם פילוסופיית ה-Manual Override שלך.

### 2.2 ייבוא Wishlist מ-Google Maps
- אין API לרשימות שמורות. המסלול המעשי: ייצוא דרך Google Takeout (CSV) + העשרה נקודתית דרך Places API.
- **המלצה:** לתכנן זאת כ"ייבוא תקופתי" ולא כסנכרון חי, ולבנות ממשק ייבוא סלחני (גרירת קובץ CSV על המפה).

### 2.3 Auto-Balancing תקציבי
- האלגוריתם עצמו פשוט; הבעיה היא **איכות הנתונים** — אין מקור חי למחירי לינה (שוב, אין Airbnb API). איזון אוטומטי מול הערכות מחיר ייצר החלטות שגויות בביטחון מלא.
- **המלצה:** במקום איזון אוטונומי — **Budget Guardrail**: רמזור שבועי מול יעד תקציב + הצעות ("3 לילות House Sitting באזור X יחזירו אותך לירוק"). ההחלטה תמיד אצלך.

## 3. מפה ו-Offline

- **Leaflet לעומת Mapbox:** ההמלצה שלי היא דווקא אופציה שלישית — **MapLibre GL JS + PMTiles (Protomaps)**. Mapbox יקר וה-Offline שלו בדפדפן מוגבל; Leaflet מבוסס Raster ומסורבל ל-Offline. MapLibre חינמי (ללא Token), ו-PMTiles מאפשר קובץ אריחים וקטורי אחד לאזור שלם שנשמר מקומית.
- **מגבלת אחסון ב-PWA:** כל אירופה לא תיכנס לטלפון. הפתרון: הורדת Extracts אזוריים לפי תוכנית המסלול (המדינה הנוכחית + הבאה), עם ניהול "אזורים שמורים" בממשק.
- **iOS:** אם המכשיר הראשי הוא iPhone — חובה PWA מותקן (לא טאב בספארי) + בקשת Persistent Storage + גיבוי JSON אוטומטי ל-Drive, כי ספארי עלול לפנות אחסון.
- **שכבת LEZ:** אין מקור נתונים חינמי ונקי לאזורי פליטה מופחתת באירופה (urbanaccessregulations.eu אינו API). לתכנן זאת כשכבת GeoJSON מתוחזקת ידנית לאזורים הרלוונטיים במסלול. עבור לנד קרוזר (סביר דיזל) זו שכבה קריטית — טוב שזוהתה מראש.

## 4. Make.com

מתאים לתהליכים אסינכרוניים אמיתיים (יצירת טיוטות פנייה, התראות), אבל **אסור שיהיה כותב שני על נתוני הליבה**. אם נשאר בתמונה: Make כותב רק ל"תיבת דואר נכנס" ייעודית, והאפליקציה ממזגת ממנה עם אישור ידני. צווארי הבקבוק הצפויים בו: מגבלות Operations בתמחור, ודיבוג אטום כשתרחיש נכשל באמצע אירופה בלי אינטרנט יציב.

               │
   ┌───────────┼────────────────┬──────────────┐
   ▼           ▼                ▼              ▼
Calendar    Drive           Sheets         Make.com
(השתקפות,  (גיבוי JSON)    (ייצוא דוחות,  (טיוטות פנייה,
 דו-כיווני                  חד-כיווני)     inbox בלבד)
 מבוקר)

## 6. סדר בנייה מומלץ

| שלב | תוכן | הערה |
|-----|------|------|
| MVP | מפה + מסד מקומי + הזנה ועריכה ידנית + דחיפה חד-כיוונית ליומן | זה כבר מוצר שימושי לדרך |
| שלב 2 | קריאה חזרה מהיומן (sync tokens), ייבוא Wishlist, שכבות LEZ/P+R | |
| שלב 3 | Budget Guardrail, טיוטות Outreach, גיבוי אוטומטי | האוטומציות אחרונות — לא ראשונות |

העיקרון: לבנות קודם את הליבה הידנית והיציבה, ורק אז להלביש עליה אוטומציות. ההפך — לבנות אוטומציות על סנכרון לא יציב — הוא הדרך הבטוחה למערכת שלא סומכים עליה באמצע נסיעה.
